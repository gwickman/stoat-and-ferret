LEARNING 2: LRN-842
-------------------
Title: Route blocking file I/O off shared asyncio event loops
Tags: design-phase, architecture, asyncio, failure-mode, quality-gates
Summary: Synchronous file writes, large reads, and subprocess-backed post-processing inside async functions block the entire shared event loop. Move blocking sequences off-loop; do not rely on async lint rules.
Source: Sonar-remediation review, 2026-07
Applicability: HIGH
  v109 COMPLEX-1 refactoring includes API/DB/DSP hotspots (scan.py, waveform.py, qc_service.py, tts_service.py).
  These may involve subprocess calls or blocking I/O. v108 addressed this (BL-661/699/700); v109 must maintain invariant.
Recommendation: Apply to v109 during refactoring
  Ensure no new blocking I/O introduced on shared event loop during service refactoring.
  Reference this learning in design ACs for affected routers/services.

LEARNING 3: LRN-843 (CRITICAL)
-----------------------------
Title: Confine filesystem paths derived from external identifiers to their intended base
Tags: design-phase, architecture, validation, failure-mode, pattern
Summary: Any feature converting external identifier/user value into filesystem path must confine resolved path strictly under intended base. Inconsistent reuse of confinement check is how the gap reappears.
Source: Sonar-remediation review, 2026-07
Applicability: CRITICAL
  v108 closed BL-699/700 (scan path confinement + worker re-validation).
  v109 must NOT regress this during COMPLEX-1 refactoring. Particularly relevant to scan.py, asset upload, new path-handling logic.
Recommendation: MANDATORY REGRESSION GUARD
  Add to v109 design ACs: "All filesystem path operations derived from external IDs must pass path-confinement validation consistently across all call sites (no gap re-emergence)."
