# Phase 4: Preview & Playback Design

Phase 4 extends stoat-and-ferret with real-time preview generation, HLS-based playback, proxy workflow, and an AI Theater Mode for full-screen spectator viewing. The design builds on the established hybrid Python/Rust architecture: Python manages preview orchestration, HLS segment serving, and WebSocket event broadcasting; Rust adds preview filter simplification for real-time performance. Two deferred items from earlier phases (BL-086, BL-141) are resolved at the start of Phase 4. The design produces 30 backlog items (8 Small, 17 Medium, 5 Large) mapped across 4 suggested versions (v024-v027).

## Key Design Decisions

1. **HLS over libmpv**: The roadmap suggested libmpv for playback, but HLS.js is already in the tech stack and provides browser-native playback without native dependencies. HLS-first with libmpv as optional future enhancement. This eliminates a cross-platform native dependency and keeps the GUI purely web-based.

2. **Proxy-first preview**: All preview playback uses proxy files (540p/720p) rather than originals. This enables smooth playback even with complex effect chains and large source files.

3. **FFmpeg as segment generator**: HLS segments are generated by FFmpeg subprocess (consistent with existing patterns). No streaming server needed - FastAPI serves static segment files.

4. **Minimal Rust scope**: Per LRN-012, Rust is only used where it provides genuine value. Phase 4 adds one Rust module: preview filter simplification (transforming production filter chains into lighter preview variants). All other logic stays in Python.

5. **WebSocket-first progress**: Preview and playback progress uses WebSocket push (resolving BL-141 pattern) rather than polling. Consistent with existing WebSocket event infrastructure.
