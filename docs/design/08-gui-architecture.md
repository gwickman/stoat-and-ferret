# stoat-and-ferret - GUI Architecture

**Project:** stoat-and-ferret â€” AI-driven video editor with hybrid Python/Rust architecture

## Overview

This document defines the GUI architecture for the AI-driven video editor. The GUI is a **web-based interface served by the FastAPI server**, designed to evolve alongside the API milestones. It provides visibility into system operations, enables interactive testing of features as they land, and includes a dedicated **AI Theater Mode** for full-screen AI-streamed content viewing.

**Design Philosophy:**
- **API-First**: The GUI is just another client of the REST API
- **Progressive Enhancement**: Start as an operations dashboard, evolve into a full editor
- **Transparency**: Show what the system is doing (FFmpeg commands, Rust core activity)
- **AI-Native**: Designed assuming an AI often drives operations

### Client-Side Routes

The React app uses `react-router-dom` with `BrowserRouter` rooted at `/gui`:

| URL | Page Component | Description |
|-----|----------------|-------------|
| `/gui/` | `DashboardPage` | Operations dashboard with health and metrics |
| `/gui/library` | `LibraryPage` | Video library browser with search |
| `/gui/projects` | `ProjectsPage` | Project list and management |
| `/gui/effects` | `EffectsPage` | Effect Workshop with full CRUD workflow |

Navigation tabs are dynamically shown/hidden based on backend endpoint availability checks.

**Known Limitation â€” SPA Fallback:** Direct URL access to client-side routes (e.g., typing `/gui/effects` in the browser address bar) may return 404 since `StaticFiles` serves files, not SPA routes. **Workaround:** Navigate to `/gui` first, then use in-app links. The React Router handles routing correctly once the SPA is loaded.

## AI Theater Mode

A dedicated full-screen mode for watching AI-driven video generation and editing.

### Purpose

Theater Mode transforms the user into a **spectator** watching the AI work:
- Full-screen video canvas showing AI-generated preview
- Minimal overlay displaying AI actions and progress
- Immersive viewing of the AI's creative process
- Works during both editing (AI building the project) and rendering

### WebSocket Event Stream

The backend broadcasts real-time events for Theater Mode:

```typescript
// Event types for Theater Mode
interface AIActionEvent {
  type: "ai_action";
  action: "adding_clip" | "applying_effect" | "removing_clip" | "adjusting_timing";
  target: string;  // clip_id or effect_id
  description: string;  // Human-readable: "Adding text overlay 'Chapter 1'"
  timestamp: string;
}

interface RenderProgressEvent {
  type: "render_progress";
  job_id: string;
  progress: number;  // 0.0 - 1.0
  current_frame: number;
  total_frames: number;
  eta_seconds: number;
  current_operation: string;  // "encoding_frame", "applying_filter", etc.
}

interface PreviewReadyEvent {
  type: "preview_ready";
  stream_url: string;  // HLS manifest URL
  duration: number;
  project_id: string;
}

interface PreviewUpdateEvent {
  type: "preview_update";
  position: number;  // Current playhead position
  buffered: number;  // Buffered amount
}
```

### Streaming Options

| Method | Use Case | Latency | Complexity |
|--------|----------|---------|------------|
| **HLS** | Rendered preview | ~2-5s | Low |
| **WebSocket + JPEG** | Live editing preview | ~100ms | Medium |
| **WebRTC** | Real-time collaboration | ~50ms | High |
| **Progressive MP4** | Final output viewing | Varies | Very Low |

**Recommended:** HLS for most cases, with WebSocket frame streaming for live editing feedback.

## Build & Deployment

### Development

```bash
# Terminal 1: Backend
uvicorn src.main:app --reload

# Terminal 2: Frontend (with hot reload)
cd gui
npm run dev
```

Vite proxies API requests to the backend during development.

### Production Build

```bash
# Build frontend
cd gui
npm run build

# Built files go to gui/dist/
# FastAPI serves them from /gui/*
```

### Docker Integration

```dockerfile
# Stage: Build frontend
FROM node:20 AS gui-builder
WORKDIR /gui
COPY gui/package*.json ./
RUN npm ci
COPY gui/ ./
RUN npm run build

# Stage: Production (add to existing Dockerfile)
COPY --from=gui-builder /gui/dist /app/static/gui
```

## Performance Considerations

### Frontend Performance

| Concern | Solution |
|---------|----------|
| Large video lists | Virtual scrolling |
| Timeline rendering | Canvas-based rendering |
| Real-time updates | Debounced WebSocket handlers |
| Bundle size | Code splitting by route |

### Backend Performance

| Concern | Solution |
|---------|----------|
| WebSocket scaling | Connection pooling, heartbeat |
| Preview streaming | Proxy file caching |
| Concurrent renders | Queue with limits |

## Quality Requirements

### GUI-Specific Testing

- [ ] Unit tests for components (Vitest)
- [ ] Integration tests for API calls (MSW)
- [ ] E2E tests for critical flows (Playwright)
- [ ] Accessibility audit (axe-core)

### Backend Integration Testing

The GUI relies on the REST API, which is validated by the black box testing harness (see 07-quality-architecture.md). This ensures:

- [ ] All API endpoints the GUI calls are covered by black box tests
- [ ] WebSocket events are validated in integration tests
- [ ] Error responses are structured and predictable
- [ ] Real Rust core is used (not mocked) in all backend tests

**GUI + Backend Integration:**
```typescript
// Frontend can rely on API contracts validated by black box tests
// If black box tests pass, API behaves as documented

// Example: Video scan workflow
const response = await fetch('/api/v1/videos/scan', {
  method: 'POST',
  body: JSON.stringify({ path: '/media' })
});
// Black box tests verify this returns { scanned, new, updated, errors }
```
