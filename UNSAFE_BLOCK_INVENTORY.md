## Rationale

The Rust core's responsibilities are pure compute:

- Filter-graph construction (`ffmpeg/`)
- Timeline math (`timeline/`)
- Input sanitization (`sanitize/`)
- Layout / preset / clip / batch math
- Render plan and progress parsing
- Schema and version metadata

None of these tasks require FFI to non-Rust libraries, manual memory
layout, or raw pointer arithmetic. The PyO3 boundary itself is
generated by `#[pymethods]` and `#[pyclass]` macros, which expand to
safe Rust that calls into the PyO3 runtime — no `unsafe` is required at
the user-author level.

External crate dependencies that contain `unsafe` (e.g., `pyo3`,
`regex`, the standard library) are scoped to those crates' own audits;
they are exercised through their safe public APIs from this crate.

## Re-Audit Trigger

If a future feature introduces any of the following, this inventory
must be regenerated and each `unsafe` block annotated with its safety
invariant:

- FFI to C/C++ libraries beyond what PyO3 already provides
- Manual `Vec`/`String` capacity manipulation via raw pointers
- `mem::transmute`, `Box::from_raw`, or `Pin`/`Unpin` projection that
  cannot be expressed safely
- SIMD intrinsics from `std::arch`

The CI lint (`cargo clippy -- -D warnings`) does not currently enforce
`#![deny(unsafe_code)]` at the crate root, so adding `unsafe` is not
syntactically blocked. Reviewers should treat the introduction of any
`unsafe` as a security-impacting change and require explicit
justification in the PR.
